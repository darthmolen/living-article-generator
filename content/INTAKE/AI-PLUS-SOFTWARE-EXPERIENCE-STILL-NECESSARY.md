# AI + Software Experience: Why Human Expertise Still Matters

AI models are task completers, not architects. Given a requirement like "add Postgres persistence," a model will produce working code that satisfies the immediate need -- repository classes, write-through guards, migration scripts. It will pass tests. It will ship. But it optimizes for *completing the task*, not for the system's long-term health.

The model doesn't care whether you'll need Redis for cache scaling in six months, or whether another service will consume the same state, or whether the orchestrator is accumulating responsibilities that make every future change riskier. It doesn't recognize that `if self.repository:` scattered across seven mutation sites is the same anti-pattern it would flag in a code review. It solves the problem in front of it, and it solves it well enough to look correct.

This is where human software engineering experience earns its keep. A senior engineer looks at the AI's plan and sees the cascade: direct repository injection today means scattered persistence guards, which means cache and DB can drift, which means recovery logic is bolt-on, which means swapping the cache layer later requires touching every call site. The fix isn't more code -- it's a service abstraction that the AI didn't propose because it wasn't part of the stated requirement. The human recognized the pattern, named it, and redirected before a single line was written.

The lesson isn't that AI code is bad. It's that AI code is fit-for-purpose -- and "purpose" is narrowly defined by the prompt. Without someone who has lived through the consequences of tight coupling, who has been the one refactoring scattered guards at 2 AM because a new feature touched twelve files instead of one, the AI's output becomes the foundation for a fragile system that only gets harder to change. Human expertise turns "working code" into "sustainable code," and that distinction is the difference between a prototype and a product.
